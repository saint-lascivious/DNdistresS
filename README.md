## description

```text
Topic: description

DNdistresS continuously (or time-bounded) sends DNS a configurable volume of 
queries of any type from a prepared domain top domains list and/or custom list,
using token-bucket scheduling and a worker pool.

Whether it's a trickle of queries or a flood, in bursts or at a constant rate,
on demand or as a periodic background task, with dig or nslookup, DNdistresS
is designed to generate controlled DNS traffic for cache priming, testing,
benchmarking, or otherwise simulating workloads against resolvers.

Key properties:
  - Deterministic rate shaping via qps + burst
  - Parallel execution via maximum workers
  - Flexible source handling (download/cache/file/custom list)
  - Resolver targeting (local/remote, IPv4/IPv6, optional port override)
  - Query tool selection (dig/nslookup) and RR type selection

DNdistresS is an impractically complex resolver workload tool for controlled DNS
traffic generation.
```

## runtime

```text
Topic: runtime

Data path:
  domain source(s) -> parser -> deduplication -> top-N -> dispatch -> workers

Primary controls:
  -q/--qps          tokens per second
  -B/--burst        burst ceiling
  -m/--maximum      worker count
  -D/--duration     runtime seconds (0 = forever)
  -o/--output       quiet|answer

Secondary controls:
  --force-maximum   override default max worker clamp (128)

Timer guard:
  When running under dndistress.service from dndistress.timer,
  _DURATION is clamped to the timer period (OnUnitActiveSec) to prevent
  runaway execution. This means that if you set _DURATION to a value longer
  than the timer period, it will be automatically reduced to match the timer
  period.
  Setting _DURATION to 0 is also clamped by not treating it as infinite when
  running under the timer.
```

## domains

```text
Topic: domains

Domain parsing/validation policy:

  DNdistresS uses a pragmatic DNS hostname filter for workload hygiene.
  It is intentionally not full RFC/IDN validation.

Current behavior (summary):
  - Lowercases input and trims one trailing dot.
  - Rejects empty values, consecutive dots, overlong names (>253),
    overlong labels (>63), non [a-z0-9-] label characters, and labels
    that start/end with '-'.

Rationale:
  Keep parsing fast, predictable, and operationally useful for resolver
  workload generation, without claiming strict standards completeness.
```

## deny-any

```text
Topic: deny-any (--deny-any / --allow-any)

Meaning:
  Controls whether querying RR type ANY (ID 255) is permitted.

Default: --deny-any (ANY queries are rejected)

--deny-any    Reject _TYPE=ANY (default behavior).
--allow-any   Permit _TYPE=ANY queries.

Note:
  ANY (RR type 255) is non-standard for most resolver workloads and
  may be refused or rate-limited by many resolvers. Enable explicitly
  with --allow-any if you know your target resolver supports it and
  you actually have a legitimate use case for generating ANY queries.
```
